Overview

Segments Manager is a tool for managing network VLAN segments across multiple sites. It supports automatic allocation via the API, a read-only dashboard for visibility, and shared segments where multiple clusters can use the same VLAN.

Key features

  • Multi-site support — manage VLANs across different sites (site1, site2, site3).
  • DHCP configuration — enable/disable DHCP per segment.
  • MongoDB backend — segments stored in MongoDB with atomic allocation.
  • Flexible subnets — support for a range of subnet sizes.
  • Automatic allocation — find and allocate available segments programmatically.
  • Shared segments — multiple clusters can share the same VLAN segment.
  • Read-only dashboard — the web UI is a live view; creation, allocation, and edits happen via the API.
  • Export — download data as CSV or Excel.
  • Real-time search — search by cluster, EPG, VLAN, or segment.

Core concepts

Sites

The system manages three sites, each with a required IP prefix:

  • site1 — segments must start with 192.x.x.x
  • site2 — segments must start with 193.x.x.x
  • site3 — segments must start with 194.x.x.x

Segment types

Every segment has a type, one of MCE, INVENTORY, HC, PXE. Optional — defaults to HC if omitted when creating a segment.

Segment states

Every segment carries a status field. Lifecycle: Locked → Available → Allocated → Available. Locking is one-way — once unlocked, a segment can never become locked again via the API.

  • Locked — the default status for a newly-created segment. Firewall rules for the segment have not been opened yet, so it's excluded from automatic allocation. An external service unlocks it via POST /api/segments/unlock (by CIDR) once provisioning is complete. While firewall requests await approval, the orchestrator publishes their ids via PUT /api/segments/connectivity-requests; the table then shows a Requests ID button beside the status badge (click for a popover with the pending ids) until the requests complete.
  • Available — unlocked and not allocated to any cluster (including previously-released segments; the released flag records that history).
  • Allocated — assigned to one or more clusters.

Allocation logic

When allocating a segment for a cluster:

  1. Check if the cluster already has a segment at the site.
  2. Check if the cluster is part of a shared segment.
  3. If not found, atomically allocate the first segment with status Available.
  4. If no Available segments exist at the site, return an error.

Interface guide

Main dashboard

The web UI is a read-only visualization of the segments stored in MongoDB. It shows per-site statistics and the full segments table, with search, status filters (All / Available / Allocated), and a site filter. When a segment has firewall requests awaiting approval, a Requests ID button appears beside its status badge — click it for a small popover listing the pending request ids; it disappears automatically once the requests complete.

Exporting data

  • CSV / Excel — download the currently filtered segments list.
  • Stats — download per-site utilization statistics as CSV.

Creating, allocating, editing, or deleting segments

These operations aren't available from the UI. Use the REST API directly — see the API reference and examples below.

Shared segments

Shared segments allow multiple clusters to use the same VLAN, which is useful for clusters that need to communicate on the same network segment.

Creating shared segments

Use the API to set multiple cluster names on a segment:

  1. Call PUT /api/segments/clusters
  2. Set segment to the segment's CIDR and cluster_names to a comma-separated list, e.g. cluster1,cluster2,cluster3

Managing shared segments

  • Add clusters — call the endpoint again with more cluster names.
  • Remove clusters — call the endpoint again with specific cluster names removed.
  • Release segment — send an empty cluster_names value to release all allocations.

Example scenarios

  • Single to shared: cluster1 → cluster1,cluster2
  • Add to shared: cluster1,cluster2 → cluster1,cluster2,cluster3
  • Remove from shared: cluster1,cluster2,cluster3 → cluster1,cluster2
  • Release all: cluster1,cluster2 → (empty)
Note: When using the API to release a single cluster from a shared segment, only that specific cluster is removed. Other clusters continue to use the segment.

API reference

All endpoints use JSON and are prefixed with /api. Single-segment operations are identified by the segment's CIDR value (its natural key — unique and immutable), never by a database id: reads and deletes take it as a ?segment= query parameter, writes carry it in the request body.

Authentication: GET requests are open. Every write (POST/PUT/PATCH/DELETE) must send the API token as a Authorization: Bearer <token> header. The token is the only credential — there is no username/password login. Missing or invalid tokens return 401.

Segment management

GET /api/segments — Get all segments
Query parameters: - site (optional): Filter by site - status (optional): Filter by lifecycle status (Locked/Available/Allocated) - type (optional): Filter by segment type (MCE/INVENTORY/HC/PXE)
POST /api/segments — Create a new segment
{ "type": "MCE", "site": "site1", "vlan_id": 100, "epg_name": "EPG_PROD_01", "segment": "192.168.1.0/24", "dhcp": true }
Note: New segments always start with status: "Locked" and are excluded from allocation until unlocked — see the unlock endpoints below. Locking is one-way; there is no endpoint to re-lock a segment. type is optional and defaults to HC if omitted.
POST /api/segments/unlock — Unlock a segment by CIDR (status Locked → Available)
{ "segment": "192.168.1.0/24" }
PUT /api/segments/connectivity-requests — Set the pending connectivity request ids shown in the UI (empty list clears the display)
{ "segment": "192.168.1.0/24", "request_ids": [123456, 654321] }
GET /api/segments/by-segment — Get a specific segment by CIDR
Query parameters: - segment (required): The segment CIDR, e.g. ?segment=192.168.1.0/24
PATCH /api/segments — Update a segment's DHCP flag (the only mutable field)
{ "segment": "192.168.1.0/24", "dhcp": true }
PUT /api/segments/clusters — Update cluster assignment (empty releases the segment)
{ "segment": "192.168.1.0/24", "cluster_names": "cluster1,cluster2,cluster3" }
DELETE /api/segments — Delete a segment by CIDR (only if not allocated)
Query parameters: - segment (required): The segment CIDR, e.g. ?segment=192.168.1.0/24

Segment allocation

POST /api/allocate-segment — Allocate a segment for a cluster
{ "cluster_name": "my-cluster", "site": "site1" }
POST /api/release-segment — Release a cluster's segment
{ "cluster_name": "my-cluster", "site": "site1" }

Statistics & information

GET /api/stats — Per-site statistics
GET /api/sites — Available sites
GET /api/health — Health check with MongoDB connectivity

Search & export

GET /api/segments/search — Search segments
Query parameters: - q: Search query (cluster, EPG, VLAN ID, or segment) - site (optional): Filter by site - allocated (optional): Filter by allocation status
GET /api/export/segments/csv — Export segments as CSV
GET /api/export/segments/excel — Export segments as Excel

Examples

Create and allocate a segment

# 1. Create a new segment (starts locked) curl -H "Authorization: Bearer $API_TOKEN" -X POST "http://localhost:8000/api/segments" \ -H "Content-Type: application/json" \ -d '{ "type": "MCE", "site": "site1", "vlan_id": 150, "epg_name": "EPG_WEB", "segment": "192.168.15.0/24", "dhcp": true }' # -> {"message": "Segment created", "id": "..."} # 2. Unlock it once firewall rules are open (Locked segments can't be allocated) curl -H "Authorization: Bearer $API_TOKEN" -X POST "http://localhost:8000/api/segments/unlock" \ -H "Content-Type: application/json" \ -d '{"segment": "192.168.15.0/24"}' # 3. Allocate it to a cluster curl -H "Authorization: Bearer $API_TOKEN" -X POST "http://localhost:8000/api/allocate-segment" \ -H "Content-Type: application/json" \ -d '{ "cluster_name": "web-cluster", "site": "site1" }'

Create a shared segment

# 1. First allocate to one cluster curl -X POST "http://localhost:8000/api/allocate-segment" \ -H "Authorization: Bearer $API_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "cluster_name": "frontend", "site": "site1" }' # 2. Update to share with multiple clusters (keyed by the segment CIDR) curl -X PUT "http://localhost:8000/api/segments/clusters" \ -H "Authorization: Bearer $API_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "segment": "192.168.15.0/24", "cluster_names": "frontend,backend,database" }'

Search segments

# Search for segments containing "web" curl "http://localhost:8000/api/segments/search?q=web" # Search allocated segments in site1 curl "http://localhost:8000/api/segments/search?q=cluster&site=site1&allocated=true"

Troubleshooting

VLAN ID already exists

Error: "VLAN X already exists for site Y"

Solution: Each VLAN ID must be unique within a site. Choose a different VLAN ID or edit the existing segment.

Invalid IP prefix

Error: "Invalid IP prefix for site. Expected to start with X"

Solution: Ensure your segment starts with the correct prefix — site1: 192.x.x.x, site2: 193.x.x.x, site3: 194.x.x.x.

No available segments

Error: "No available segments"

Solution: Create new segments for the site or release existing allocated segments.

Cannot delete allocated segment

Error: Segment deletion fails

Solution: Release the segment first, then delete it.

Best practices

  • Use descriptive EPG names.
  • Plan your VLAN ID ranges per site to avoid conflicts.
  • Use shared segments sparingly and document which clusters share them.
  • Regularly export data for backup purposes.
  • Monitor the statistics to track utilization.
